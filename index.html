Overview

02 — PASSIVE

Architecture

How the pipeline works

A scout agent runs first on every query. Its findings determine what happens next: either conditional routing to pre-built agents for known signal patterns, or dynamic agent spawning for novel situations. Hover each node to learn more.

💬
TELEGRAM INTERFACE
User input · Command routing · Alert delivery
The entry point for all queries. Accepts /signal, /watch, /unwatch, /watchlist, /watched, and /trigger commands. Also receives autonomous alerts from the watchlist monitor when signal thresholds are exceeded.
└── routes to
🧭
ORCHESTRATOR
Claude Haiku · Routing logic · Pipeline coordination
Determines workflow type and delegates to the scout. After the scout returns, decides whether to use conditional routing or trigger the supervisor for dynamic spawning. Haiku is used here because routing decisions require classification, not depth.
└── always runs first
🔍
SCOUT AGENT
Claude Sonnet · Full data gather · Signal profiling
Queries NewsAPI, Serper, Yahoo Finance, and SEC EDGAR to build a structured signal profile: funding events, leadership changes, hiring spikes, sentiment shifts, and a novelty score from 0 to 10. The novelty score determines the routing decision. If signals are contradictory, cross-domain, or structurally unusual, the score is high.
└── novelty score determines path
NOVELTY SCORE < 7 · Conditional Routing
💰 Funding Agent: round size, investor quality, competitive signal
📈 Organic Growth Agent: hiring spike without funding (the meaningful signal)
👔 Market Impact Agent: leadership change at a public company
🗣️ Reputation Agent: sentiment source, severity, recovery pattern
NOVELTY SCORE ≥ 7 · Dynamic Agent Spawning
⚡ Supervisor reads raw scout findings
✍️ Writes purpose-built agent instructions at runtime
🤖 Spawns agent configured for exact situation found
🔬 Agent didn't exist before this query
└── all outputs flow to
📋
SYNTHESIS AGENT
Claude Sonnet · Structured brief · Recommended action
Receives all agent outputs, produces a structured brief with labeled sections per agent, a recommended action (Monitor / Research Further / High Priority), and a market note if the company is publicly traded. If dynamic spawning was used, the output is flagged with ⚡ Complex Situation Detected.

Design Decisions

Why it's built this way

Every architectural choice was made deliberately. Here's the reasoning behind each one.

🔀
Conditional Routing for Known Patterns
For predictable signal categories like funding events, hiring spikes, leadership changes, and sentiment shifts, the orchestrator routes to pre-built specialized agents. These categories are stable enough that a well-configured agent handles them consistently across any company.
Tradeoff Considered Spawning for everything would be simpler to implement uniformly. Pre-built agents win on speed, cost, and consistency for known patterns. Spawning is reserved for where it actually adds value.
⚡
Dynamic Agent Spawning for Edge Cases
When the scout returns a novelty score of 7 or above due to contradictory signals, cross-domain events, or structurally unusual situations, a supervisor agent reads the raw findings and writes a purpose-built agent at runtime. That agent does not exist until that moment. Inflection AI is the example: a simultaneous acqui-hire and continuing entity does not fit any pre-built routing pattern.
Tradeoff Considered Spawning is slower and more expensive than routing. It only triggers when the signal is novel enough that a generic agent would miss the nuance or produce conflicting outputs.
🏗️
Scout-First Architecture
The scout runs before any routing decision is made. This means the system cannot know what it needs until it looks, which is the correct model for intelligence work. The scout's novelty score is load-bearing: it is what makes the routing decision non-trivial and what justifies the orchestrator existing at all.
Tradeoff Considered Running the scout first adds latency to every query. Routing based on the company name alone would miss the actual signal and reduce the orchestrator to a simple switch statement.
🧠
Two Models, Two Jobs
Claude Haiku handles the orchestrator, watchlist lightweight scout, and routing decisions where speed and cost matter more than depth. Claude Sonnet handles the full scout, all analysis agents, dynamic spawning, and synthesis where output quality determines the value of the entire pipeline.
Tradeoff Considered Using Sonnet everywhere would produce marginally better routing decisions at 3x the cost. Haiku classification is accurate enough for binary routing logic and keeps the always-on watchlist monitor cheap to run.
📡
Signal-Driven Watchlist Monitor
The watchlist runs a cheap Haiku-powered lightweight scout every 6 hours. It only returns a score with no analysis. If the score is below 7, nothing happens. If it hits 7 or above, the full Sonnet pipeline fires. This means expensive analysis only runs when something meaningful is actually detected.
Tradeoff Considered A time-driven monitor (run full analysis every 6 hours regardless) would cost ~10x more and flood the user with low-signal output. Signal-driven monitoring is both more useful and more efficient.
💬
Telegram as the Interface
An intelligence tool's value is in being alerted when something matters, not in going somewhere to check. Telegram is already on your phone, sends push notifications, and its command structure maps naturally onto interacting with an agent. Building a web frontend would have added days of work for no functional gain.
Tradeoff Considered A web dashboard would allow richer data visualization. For a personal intelligence tool where the primary value is timely alerting, Telegram delivers that at zero infrastructure cost.
