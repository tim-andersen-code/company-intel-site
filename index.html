Overview

Two ways to use it

Query any company on demand, or add companies to a persistent watchlist that autonomously alerts you when a signal threshold is crossed. Not on a timer, but when something meaningful is actually detected.

01 — ON DEMAND
Signal Query
Send /signal [company] and the full intelligence pipeline runs immediately. The scout agent gathers data, the orchestrator routes to the right agents, and a structured brief is delivered in 30 to 90 seconds. Any company works.
02 — PASSIVE
Watchlist Monitor
Add companies with /watch and a lightweight background scout checks each one every 6 hours. When a company's signal score hits 6 or above, the full pipeline fires automatically and an alert is sent without any manual query needed.

Under the Hood

How urgency is measured

Every query returns a signal score from 0 to 10. That score is the load-bearing element of the entire system — it determines routing, triggers alerts, and decides whether a standard agent or a purpose-built one runs.

Signal Score Scale

0246810
0 – 5 · Quiet
Nothing meaningful detected. Normal operations. No alert fires.
6 – 7 · Active
Meaningful signal detected. Full pipeline fires. Standard routing to pre-built agents.
8 – 10 · Complex
Contradictory or novel signals. Dynamic agent spawned at runtime for exact situation found.

Four Scoring Dimensions

📊
Volume
How many distinct signal types were detected across funding, hiring, leadership, sentiment, and market data.
⏱️
Recency
How recently the signals occurred. A leadership change from last week scores higher than one from six months ago.
⚡
Contradiction
Whether signals conflict with each other. Strong revenue alongside accelerating executive departures is a high-contradiction signal.
🔬
Novelty
How unusual the situation is relative to normal activity for a company of that type, size, and sector.

Design Decisions

Why it's built this way

Every architectural choice was made deliberately. Here's the reasoning behind each one.

🔀Conditional Routing for Known Patterns
For predictable signal categories like funding events, hiring spikes, leadership changes, and sentiment shifts, the orchestrator routes to pre-built specialized agents. These categories are stable enough that a well-configured agent handles them consistently across any company.
Tradeoff Considered Spawning for everything would be simpler to implement uniformly. Pre-built agents win on speed, cost, and consistency for known patterns. Spawning is reserved for where it actually adds value.
⚡Dynamic Agent Spawning for Edge Cases
When the scout returns a novelty score of 6 or above due to contradictory signals, cross-domain events, or structurally unusual situations, a supervisor agent reads the raw findings and writes a purpose-built agent at runtime. That agent does not exist until that moment. Inflection AI is the example: a simultaneous acqui-hire and continuing entity does not fit any pre-built routing pattern.
Tradeoff Considered Spawning is slower and more expensive than routing. It only triggers when the signal is novel enough that a generic agent would miss the nuance or produce conflicting outputs.
🏗️Scout-First Architecture
The scout runs before any routing decision is made. This means the system cannot know what it needs until it looks, which is the correct model for intelligence work. The scout's novelty score is load-bearing: it is what makes the routing decision non-trivial and what justifies the orchestrator existing at all.
Tradeoff Considered Running the scout first adds latency to every query. Routing based on the company name alone would miss the actual signal and reduce the orchestrator to a simple switch statement.
🧠Two Models, Two Jobs
Claude Haiku handles the orchestrator, watchlist lightweight scout, and routing decisions where speed and cost matter more than depth. Claude Sonnet handles the full scout, all analysis agents, dynamic spawning, and synthesis where output quality determines the value of the entire pipeline.
Tradeoff Considered Using Sonnet everywhere would produce marginally better routing decisions at 3x the cost. Haiku classification is accurate enough for binary routing logic and keeps the always-on watchlist monitor cheap to run.
📡Signal-Driven Watchlist Monitor
The watchlist runs a cheap Haiku-powered lightweight scout every 6 hours. It only returns a score with no analysis. If the score is below 6, nothing happens. If it hits 6 or above, the full Sonnet pipeline fires. This means expensive analysis only runs when something meaningful is actually detected.
Tradeoff Considered A time-driven monitor (run full analysis every 6 hours regardless) would cost ~10x more and flood the user with low-signal output. Signal-driven monitoring is both more useful and more efficient.
💬Telegram as the Interface
An intelligence tool's value is in being alerted when something matters, not in going somewhere to check. Telegram is already on your phone, sends push notifications, and its command structure maps naturally onto interacting with an agent. Building a web frontend would have added days of work for no functional gain.
Tradeoff Considered A web dashboard would allow richer data visualization. For a personal intelligence tool where the primary value is timely alerting, Telegram delivers that at zero infrastructure cost.
